[[section-runtime-view]]
== Vista de Ejecución

=== Escenario 1: Registro de un Nuevo Usuario
Este escenario describe cómo un usuario crea una cuenta. Es un proceso crítico de escritura donde la Webapp captura los datos y el servicio de usuarios asegura la persistencia.

* *Interacción*: El usuario utiliza el formulario identificado con selectores únicos (como `login-username`) para garantizar la trazabilidad en las pruebas E2E.
* *Seguridad*: El Users Service cifra la contraseña antes de almacenarla en MongoDB.

[plantuml,"Registration_Sequence",png]
----
actor Usuario
participant "Webapp (React)" as Webapp
participant "Users Service (Node)" as Users
database "MongoDB" as DB

Usuario -> Webapp: Introduce nuevos datos de cuenta
activate Webapp
Webapp -> Users: POST /adduser (username, password...)
activate Users
Users -> DB: save(newUser)
note right of Users: Cifrado de password \ny validación de esquema
DB --> Users: 201 Created
Users --> Webapp: 200 OK (Registro exitoso)
deactivate Users
Webapp --> Usuario: Redirección automática al Login
deactivate Webapp
----

=== Escenario 2: Inicio de Sesión (Autenticación)
Describe el acceso de un usuario ya existente. Este flujo es el que se valida principalmente en los tests funcionales de la aplicación.

* *API*: El Users Service expone un endpoint REST en el puerto 3000.
* *Resultado*: Se genera un token JWT que la Webapp almacenará para futuras peticiones.

[plantuml,"Login_Sequence",png]
----
actor Usuario
participant "Webapp (React)" as Webapp
participant "Users Service (Node)" as Users

Usuario -> Webapp: Introduce credenciales (#login-username)
activate Webapp
Webapp -> Users: POST /login (user, pass)
activate Users
note right of Users: Valida credenciales \ncontra MongoDB
Users --> Webapp: 200 OK (JWT + Datos de sesión)
deactivate Users
Webapp --> Usuario: Acceso concedido al Menú Principal
deactivate Webapp
----

=== Escenario 3: Inicialización de Partida y Selección de Bot
El motor de Rust mantiene un **estado persistente** mediante un `AppState` que utiliza `Arc<Mutex<String>>` para gestionar la concurrencia de forma segura.

* *Lógica de Negocio*: La dificultad seleccionada filtra la estrategia de la IA (Edge, Attacker, etc.).
* *Persistencia en Memoria*: El bot elegido queda fijado en el estado del servidor hasta el fin de la partida.

[plantuml,"Game_Init_Sequence",png]
----
actor Usuario
participant "Webapp (React)" as Webapp
participant "Users Service (Node)" as Users
participant "Gamey Engine (Rust)" as Gamey

Usuario -> Webapp: Selecciona Tamaño y Dificultad
activate Webapp
Webapp -> Users: POST /reset {size, difficulty} (JWT)
activate Users
note right of Users: Valida sesión y \nredirige petición a Gamey
Users -> Gamey: POST /reset {username, size, difficulty}
activate Gamey
note right of Gamey: 1. Reinicia instancia GameY\n2. Filtra bots por nivel\n3. Fija **active_bot**
Gamey --> Users: 200 OK (Tablero inicial)
deactivate Gamey
Users --> Webapp: 200 OK (Sesión de juego creada)
deactivate Users
Webapp --> Usuario: Renderiza tablero vacío
deactivate Webapp
----

=== Escenario 4: Turno de Juego y Persistencia de Resultados
Este escenario ilustra el ciclo de vida de un movimiento. El motor de Rust calcula la respuesta de la IA y, si detecta el fin del juego, persiste el récord.

* *Inteligencia*: El motor delega la jugada al `active_bot` guardado previamente.
* *Integración*: El servicio de Rust se comunica directamente con MongoDB para guardar el histórico.

[plantuml,"Game_Move_Sequence",png]
----
actor Usuario
participant "Webapp (React)" as Webapp
participant "Users Service (Node)" as Users
participant "Gamey Engine (Rust)" as Gamey
database "MongoDB Atlas (Cloud)" as Atlas

Usuario -> Webapp: Selecciona celda del tablero
activate Webapp
Webapp -> Users: POST /move {index} (JWT)
activate Users
Users -> Gamey: POST /execute-move {index, username}
activate Gamey
note right of Gamey: 1. Procesa mov. humano\n2. Ejecuta IA activa\n3. Verifica victoria

alt Si la partida termina (Winner != null)
    Gamey -> Atlas: insert_one(PartidaRecord)
    note right of Atlas: Persistencia en clúster externo
    Gamey --> Users: WinnerData + YEN State
    Users -> Atlas: updateOne({username}, {$inc: {totalScore: XP}})
    note right of Atlas: Incremento atómico de XP
else Partida continúa
    Gamey --> Users: 200 OK (Estado YEN)
end
deactivate Gamey

Users --> Webapp: 200 OK (Estado actualizado + Winner?)
deactivate Users
Webapp -> Webapp: Actualiza UI (Celdas y ganador)
Webapp --> Usuario: Muestra resultado final
deactivate Webapp
----

=== Escenario 5: Partida multijugador en tiempo real

El modo multijugador usa Socket.IO para la coordinacion en tiempo real y conserva la validacion de reglas en GameY. El navegador no llama directamente a Rust: se conecta al gateway de `users`, que autentica el socket con JWT y reenvia a GameY solo las operaciones de tablero.

[plantuml,"PVP_Sequence",png]
----
actor "Jugador A" as A
actor "Jugador B" as B
participant "Webapp" as Webapp
participant "Users Service\n(Socket.IO)" as Users
participant "GameY Engine" as Gamey

autonumber
A -> Webapp: Reta a un amigo conectado
Webapp -> Users: challenge_player(opponentUsername, boardSize)
Users -> B: challenge_player(challengeId)
B -> Users: accept_challenge(challengeId)
Users -> Gamey: POST /pvp/reset {match_id, size, players}
Gamey --> Users: board + next_turn
Users -> A: sync_board(matchId, board, currentTurn)
Users -> B: sync_board(matchId, board, currentTurn)
A -> Users: game_move(matchId, cellIndex)
Users -> Gamey: POST /pvp/move {match_id, player, index}
Gamey --> Users: board + winner? + next_turn
Users -> A: sync_board(...)
Users -> B: sync_board(...)
----

Si un jugador abandona o pierde la conexion, `socketHandler` marca la partida como abandonada y notifica `player_disconnected` a la sala. El estado PvP activo vive en memoria tanto en `users` (salas y turnos visibles) como en `gamey` (tablero validado), por lo que una caida del proceso obliga a crear una nueva partida.
